# MVP 范围

## MVP 目标

验证一个核心假设：

> 用户愿意把每天看到的有价值信息交给 Sift，因为 Sift 能把它们自动沉淀成之后真的能复用的知识库。

这个假设的关键不在“AI 总结”，而在：

- 低摩擦收集
- 自动提取和整理
- 保留来源
- 形成可浏览、可追溯、可复用的知识页

## 三阶段范围

### Phase 0：Prototype

目标：验证最小闭环。

```text
链接/截图 -> Source -> 单篇知识页
```

必须有：

- 单用户模式，暂时不做正式账号
- Web 输入入口
- 链接输入
- 图片上传
- 网页正文提取
- OpenAI Vision OCR
- 异步处理任务
- AI 生成 Source
- AI 生成 draft WikiPage
- 基础列表页
- Source 详情页
- WikiPage 详情页

暂时不做：

- 真正的移动端分享扩展
- 多用户同步
- 知识库级问答
- 自动合并多个主题页
- 复杂搜索
- 付费

Prototype 验证问题：

> 用户丢一个链接或截图进去，Sift 能不能自动生成一篇有用、可信、可读的知识页？

### Phase 1：Validated MVP

目标：验证用户是否会持续使用并复用知识。

必须有：

- 正式账号
- Inbox
- 处理状态和失败重试
- Source 保存和引用
- WikiPage 保存和编辑
- 写入时生成 embeddings
- pgvector 存储 chunk embeddings
- 粗粒度来源引用
- 基础全文搜索
- 单页或小范围问答
- 最近更新

新增验证问题：

> 用户是否会每周持续保存内容，并回到 Sift 里复用这些内容？

### Phase 2：Beta

目标：做成真正可用的早期产品。

重点：

- 手机分享入口
- 知识库级问答
- 引用片段级来源
- 相似 WikiPage 推荐
- Source 到 WikiPage 的合并建议
- 更好的搜索和筛选
- 处理队列监控
- 成本控制
- 基础付费

## 关键砍法

14 天 Prototype 不追求完整产品，只验证一条链路：

```text
保存链接或截图 -> 自动处理 -> 生成可读知识页
```

为了保护进度，Prototype 阶段明确砍掉：

- 账号系统
- iOS 原生分享扩展
- 完整 RAG
- 自动维护多篇主题页
- Claim 模型
- 高级图谱
- 团队协作

## 成功指标

Prototype：

- 链接和截图都能成功处理。
- 生成的知识页用户愿意保存。
- 用户能明确指出“这个结果比普通收藏有用”。

Validated MVP：

- 用户每周保存 10 条以上内容。
- 用户每周至少 2 次回到知识库复用内容。
- 用户能从知识库中产出文章、判断、研究笔记或行动清单。
- 用户愿意为“自动沉淀”而不是“存储空间”付费。

## 主要风险

### 异步处理

OCR、网页提取、AI 整理和 embedding 生成都可能超过普通 serverless 请求时间。Sift 从第一版开始就应该把处理建模为异步任务，而不是同步 API 请求。

### 来源信任

如果 AI 生成的回答和知识页没有来源，用户很快会不信任系统。至少从 Validated MVP 开始，问答必须带来源引用。

### WikiPage 合并

从 Source 自动合并到长期主题页，是 Sift 最难的产品逻辑之一。Prototype 阶段先让每个 Source 生成一个 draft WikiPage，后续再做相似页面推荐和合并建议。
