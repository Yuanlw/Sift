# Mobile-first Capture Roadmap

## 核心判断

Sift 当前的重点应该从“能力是否跑通”转向“用户是否愿意每天用”。

对第一批用户来说，最常见的信息入口不在 PC 页面，而在手机：

- X、微信、公众号、视频号、抖音、小红书等内容平台。
- 浏览器里的文章、报告、博客和产品页面。
- 聊天、群聊、朋友圈、评论区和 AI 对话。
- 相册里的截图、长图、白板、书页和会议投影。
- 临时想法、语音备忘和稍后整理的素材。

因此，Sift 下一阶段的产品重点不是继续增加通用 Agent 能力，而是把手机端收集体验做顺。

目标是：

```text
白天低摩擦收集 -> 按天管理 Inbox -> 后台异步处理 -> 长期按主题复用
```

## 产品原则

### 1. 先保存，不要求当场整理

用户在手机上保存资料时，通常没有时间选择分类、写标签或确认结构。

Sift 应默认接受不完整输入：

- 只有链接。
- 只有截图。
- 只有一段复制文本。
- 链接加几张截图。
- 文本加图片加一句备注。

所有输入都应该先进入 Inbox。网页抓取、OCR、转写、整理、去重、主题归并和 embedding 都放到后台处理。

### 2. Capture 表达一次真实保存行为

Capture 不应该只被理解成“链接、文本、图片三选一”。

真实保存常常是混合的：

```text
X 链接 + 复制出来的一段文字 + 两张截图 + 一句备注
```

这应该是一条 Capture，而不是四条孤立记录。

后续数据结构和 UI 应支持一条 Capture 同时包含：

- raw_url
- raw_text
- raw_attachments
- note
- source_platform
- source_app
- captured_at

### 3. 时间适合管理 Inbox，不适合替代知识结构

按天分组的价值不在“知识分类”，而在“收集管理”。

用户回看 Inbox 时，最自然的问题常常是：

- 今天刚才保存的那条在哪里？
- 昨天看到的那个观点是什么？
- 本周有哪些资料还没处理？
- 哪些截图 OCR 失败了，需要补充？

因此：

- Inbox 默认按时间分组，适合管理保存动作。
- Source 保留来源资料，适合查看证据。
- Wiki 按主题组织，适合长期复用。
- Ask / Search 跨时间、跨主题召回。

## 典型使用场景

| 场景 | 用户动作 | Sift 期望能力 |
| --- | --- | --- |
| X 平台可复制长文，带图 | 复制正文，保存图片或截图 | 粘贴长文本，多图上传，合并为一条 Capture |
| X 平台不可复制内容 | 保存链接，或截图 | 快速保存 URL，抓取失败也保留链接和截图 |
| X Thread | 保存首条链接，或复制部分内容 | 保留 thread 链接，尽量展开，失败时提示补充截图或正文 |
| 微信公众号文章 | 分享链接、复制链接、复制正文 | 移动端分享入口，正文提取失败时允许补充截图和粘贴文本 |
| 抖音 / 视频号 | 复制分享链接，截图字幕或评论 | 视频链接、截图和备注合并保存 |
| 评论区资料 | 截图评论、保存原帖链接 | OCR 评论截图，并关联原始帖子或视频链接 |
| 网页文章 / 博客 / 新闻 | 浏览器里复制链接或分享到 Sift | 自动提取标题、正文、来源和发布时间 |
| 付费墙 / 登录后文章 | 抓取失败，只能复制片段或截图 | 保留原链接、用户片段、截图和备注 |
| PDF / 研报 | 下载文件、截图关键页 | 先保存原文件或截图，后续再解析 PDF |
| 图片型资料 | 海报、图表、课程截图、朋友圈截图 | 多图上传，OCR，保留原图和识别文本 |
| 聊天记录 | 微信、飞书、群聊截图或复制 | 识别为对话资料，保留上下文和参与者线索 |
| AI 对话结果 | 复制 ChatGPT、Claude、Kimi 等回答 | 保存问题、回答、模型来源和后续备注 |
| 临时想法 | 手机里写一句话 | 极简文本入口，像发备忘录一样保存 |
| 语音想法 | 不方便打字时录音 | 先保存音频，后台转写和整理 |
| 线下资料 | 拍书页、白板、会议投影 | 图片保存，OCR 后生成可检索文本 |
| 已有收藏迁移 | 浏览器书签、微信收藏、相册截图很多 | 批量导入，按时间进入 Inbox，后台慢慢整理 |

## P5：手机优先的低摩擦收集（已完成）

### 目标

用户在手机上看到任何值得保存的东西，不需要思考分类，先一键丢进 Sift。

### 范围

- 移动端 `/inbox` 优化：打开就是快速保存。
- Inbox 默认按时间分组：今天、昨天、本周、更早。
- 一条 Capture 支持链接、文本、图片、备注混合输入。
- 支持一次上传多张截图，作为同一条资料处理。
- 保存后立刻反馈成功，不等待 OCR、网页提取或 AI 整理。
- 捕获失败、抓取失败、OCR 失败都不能丢失原始资料。
- Capture 列表清楚显示已保存、处理中、失败、已沉淀。
- 支持稍后补备注，尤其是“为什么保存这条”。

### 第一批实现清单

1. 调整 Capture 表单，让链接、文本、图片和备注可以同时提交。
2. 调整 `/api/captures` 的输入判断，不再把 Capture 强制收敛成单一类型。
3. Inbox 列表按日期分组，优先展示“今天”。
4. 移动端优化保存表单和 Capture 列表密度。
5. Capture 详情展示混合输入的完整原始材料。
6. 多图上传继续沿用私有上传目录和授权读取接口。
7. 对处理失败的 Capture 提供“补充资料”和“重试处理”的入口。

### 完成状态（2026-05-05）

- `/inbox` 已改为手机优先的 Capture Composer，支持链接、文本、图片和“为什么保存”首次备注混合提交。
- 保存链路继续保持“先落库、后处理”，不等待网页提取、OCR、向量写入或模型整理。
- Inbox 已按今天、昨天、本周、更早分组，并在默认视图中排除已忽略资料。
- 多图上传继续走私有上传目录和授权读取接口。
- Capture 详情页已展示混合输入的原始材料，并支持补备注、补充资料、重试处理和忽略。
- X 链接会优先通过 `defuddle.md` 获取单帖 Markdown，失败后再尝试 `r.jina.ai` 并清洗页面噪音；如果仍失败，则保留原始链接和用户输入，作为可重试或可忽略的资料进入整理流。

### 暂不做

- 不在 P5 做完整原生 App。
- 不要求用户保存时选择复杂分类。
- 不把按天分组当作最终知识结构。
- 不为了封闭平台强行做不稳定抓取。
- 不让保存链路等待模型处理。

## P5.5：每日整理体验（第一版已完成）

P5 做顺保存后，可以增加轻量的每日回顾。

目标不是要求用户每天整理，而是让用户容易发现当天有价值、失败或需要补充的资料。

第一版已完成能力：

- 今日收集视图。
- 待补备注视图。
- 处理失败视图。
- 处理中视图。
- 已忽略视图。
- 失败资料可重试、可补充资料后重新处理、可忽略。
- 详情页可只补备注，不必触发重新处理。

后续增强：

- 本周收集回顾。
- 一键删除明显无价值的误存资料。
- 将多条 Capture 标记为同一主题的线索。

每日整理的产品表达可以是：

> Sift 不要求你当场整理。它先按时间帮你收好，让你之后按天清理、按主题复用。

## P6：外部收藏池同步（第一版已完成）

外部收藏池同步很有价值，但应该放在 Sift 自己的收集体验做顺之后。

原因是：如果 Sift 自己的 Inbox 不好管理，同步只会把旧资料变成新的混乱。

### 第一版完成状态（2026-05-05）

- `/inbox` 增加外部收藏导入面板。
- 支持批量 URL 粘贴，一次最多导入 100 条。
- 支持浏览器书签 HTML 文件导入，读取链接标题和原始书签时间。
- 支持相册截图批量导入；每张截图作为独立 Capture 进入今日收集，并保留图片文件名、导入批次和时间元数据。
- 导入项仍走现有 Capture 保存和后台处理链路，不新建独立知识处理流程。
- 导入时自动合并同批重复链接，并跳过库里已有的同一 URL。
- 每条导入资料会在 `raw_payload` 里记录导入批次、导入来源、原始标题和导入元数据。

### 可能来源

| 来源 | 可行方式 | 判断 |
| --- | --- | --- |
| 浏览器书签 | 导入 HTML、浏览器插件、bookmarklet | 优先级高，技术可控 |
| 浏览器当前页 | 扩展或 bookmarklet 一键保存 | 适合 PC 深度研究 |
| 微信收藏 | 手动复制、截图、导出后导入 | 微信生态封闭，先做低成本方案 |
| 相册截图 | 批量上传，按拍摄日期归组 | 很贴合手机真实习惯 |
| 稍后读工具 | Readwise、Pocket、Instapaper 等导出或 API | 适合重度阅读用户 |
| X 收藏 / 点赞 | 链接、截图、手动导出优先 | API 限制和成本较高，不应第一批依赖 |
| Notion / Obsidian | Markdown 或导出文件导入 | 适合迁移历史资料 |

### 第一批同步策略

先做“导入”，再做“同步”。

第一批可以支持：

- 浏览器书签 HTML 导入。（第一版已完成）
- URL 批量粘贴。（第一版已完成）
- 相册截图批量上传。（第一版已完成）
- Markdown / 文本文件导入。

后续再考虑：

- 浏览器插件。
- iOS Shortcut。
- Android 分享入口。
- 微信收藏的半自动导入。
- 第三方稍后读工具 API。

## 后续平台能力

手机优先收集和收藏池同步完成后，再继续推进模型 provider、评测集和处理质量优化。

这些能力很重要，但它们服务的是同一个目标：

```text
让更多真实资料更低摩擦地进入 Sift，并可靠地变成可复用知识。
```
